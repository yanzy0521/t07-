# Static analysis design

## Analysis boundaries

本任务把证据分为三层：官方比赛规则、官方优化建议、Demo 实际代码。最终结论必须保持三层来源独立，再通过映射表说明它们之间的关系，避免把推荐方法误写成规则，或把文档示例误写成 Demo 已实现逻辑。

## Evidence model

每个重要结论至少记录以下信息：

1. 结论内容。
2. 来源：PDF 文档主题，或代码路径与符号。
3. 调用/数据链：定义者、写入者、读取者、最终动作消费者。
4. 可信度：已确认、高度可能、存在疑问、需要运行验证。
5. 对策略和后续修改的影响。

代码结论优先级为：完整调用链与实际数据消费 > 多处交叉印证 > 注释与命名。仅有注释或未调用定义时，不判定为当前生效行为。

## Workstreams

### 1. PDF evidence extraction

- 分别读取规则 PDF 和四份优化 PDF。
- 建立规则约束表、推荐方法表、参数/接口表和冲突表。
- 记录文档中需要与 Demo 代码核对的术语、状态、配置和动作名称。

### 2. Repository architecture reconstruction

- 枚举目录、构建/启动文件、配置、源码、测试和文档。
- 从入口向下追踪运行时对象创建、循环/回调、状态更新、策略调度和动作发布。
- 反向追踪关键动作接口和关键状态字段的所有调用者。

### 3. Strategy reconstruction

- 按比赛状态和角色建立行为矩阵。
- 对每条当前策略验证触发条件、决策分支、动作、退出条件、失败/超时处理。
- 单独记录多人协作、通信失效、状态抢占和恢复行为。

### 4. Gap and modification mapping

- 将官方推荐逐项映射到 Demo 实现。
- 对潜在修改点按策略层、公共模块、底层边界分级。
- 汇总静态证据不足或依赖运行环境的事项。

## Parallel exploration approach

分析工作可并行拆分为：

- 比赛规则与官方建议证据提取；
- 项目入口、运行时、配置和数据流；
- 策略、行为树/状态机、角色与动作调用链；
- 通信、ROS/仿真边界、测试及废弃代码检查。

主分析负责交叉核对各工作流结果，消解术语差异，并统一写入 `PROJECT_ANALYSIS.md`。任何子分析结论都必须在最终文档中回到具体 PDF 或代码证据。

## Output structure

`PROJECT_ANALYSIS.md` 使用用户指定的 16 章结构。为控制重复：规则只在规则章完整定义，后续策略章节通过规则编号引用；模块职责只在目录章定义，后续通过路径和符号引用。

## Safety and compatibility

- 只新增或更新 `PROJECT_ANALYSIS.md`，不触碰运行代码与配置。
- 读取操作不改变仓库状态。
- 若发现工作区出现非本任务造成的意外修改，立即停止并询问用户。
- 无法静态闭合的行为不做确定性判断，转入运行验证清单。
