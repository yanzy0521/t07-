# Analyze football robot demo and official documents

## Goal

通过完整静态阅读官方比赛规则、四份官方策略资料和整个虚拟足球机器人 Demo，建立一份可追溯、带可信度标记的项目与原始策略分析，为后续用户提供的策略优化方案建立可靠基线。

最终交付物为项目根目录下的 `PROJECT_ANALYSIS.md`。除该分析文件和 Trellis 任务记录外，不修改任何源代码、配置或项目行为。

## Background

- Cursor 环境无法直接编译或运行项目，本任务以代码阅读、全局搜索、调用链追踪、配置追踪和 PDF 对照为主要证据来源。
- 用户提供了以下资料：
  - `o:\Robot\赛场规则.pdf`
  - `o:\Robot\配置参数指南.pdf`
  - `o:\Robot\如何为场上选手分配角色.pdf`
  - `o:\Robot\如何优化机器人移动行为.pdf`
  - `o:\Robot\如何找到最适合的射门队员.pdf`
- 只有静态分析无法最终确认的结论才标记为“需要运行验证”。

## Requirements

### R1. 比赛规则

- 阅读完整比赛规则，整理比赛流程、比赛状态与切换、各种比赛重启、进球和犯规、球员与触球限制、裁判指令、通信及动作限制。
- 解释每条关键规则对策略判断、状态切换和动作执行的具体影响。
- 明确区分规则原文要求与基于规则推导的实现影响。

### R2. 官方优化资料

- 阅读全部四份官方优化资料，提取角色分配、进攻、防守、守门、追球、传球、射门、定位、导航、避障、通信、配置和接口结构方面有实现价值的方法。
- 区分强制规则、官方推荐、Demo 已实现、Demo 未实现和后续可选方法。
- 记录不同资料之间的冲突、适用条件或版本差异。

### R3. 全项目静态分析

- 先建立完整目录和文件职责索引，再从真实程序入口追踪初始化、配置读取、主循环、状态接收、世界数据、角色分配、策略决策、行为树或状态机、动作接口、通信和仿真边界。
- 覆盖源代码、配置、测试、构建文件、脚本、第三方或生成文件；明确哪些文件属于当前运行路径。
- 重要结论必须由实际定义、变量使用、调用点或配置消费位置支持，不能只根据命名和注释推测。

### R4. 原始策略还原

- 还原比赛开始、角色获取、普通进攻、防守、守门员、开球与其他重启、多人抢球协调、离线降级、动作失败、超时和状态变化恢复流程。
- 对关键策略采用统一格式：触发条件 -> 判断过程 -> 执行动作 -> 结束条件 -> 失败或超时处理。
- 标注未调用、重复、旧版或疑似废弃逻辑，不能将其视为当前生效策略。

### R5. 后续修改索引

- 为潜在优化点记录文件路径、类/函数/节点/配置项、当前作用、上下游关系、可能影响和修改风险。
- 区分可修改策略层、需要谨慎修改的公共模块，以及原则上不应为策略优化修改的仿真、通信和底层接口。

### R6. 证据与可信度

- 所有重要结论使用“已确认”“高度可能”“存在疑问”“需要运行验证”之一标记。
- 对引用代码给出精确路径和符号名；仅引用支撑结论所需的少量片段，不大量粘贴源码。
- 对 PDF 结论尽可能记录来源文档和章节/主题；对规则与建议保持来源隔离。

## Constraints

- 不修改源代码或配置文件，不创建策略补丁，不重构、删除或新增比赛逻辑。
- 不直接采用官方优化方案；只分析其价值及 Demo 覆盖情况。
- 不因不能编译而跳过静态分析。
- 可以创建并持续更新 `PROJECT_ANALYSIS.md`，完成后立即停止策略工作。
- 任何必须依赖真实运行时序、外部裁判/仿真器行为、网络环境或未包含依赖才能确认的事项，必须保留为验证项。
