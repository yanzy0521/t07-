# Design: 比赛规则与代码流程分析文档

## Scope and Boundaries

- 输出物是一份 Markdown 分析文档，不修改 Demo 程序源码。
- 官方比赛规则 PDF 是规则事实的最高来源。
- 官方教程 PDF、项目已有解析文档、代码注释和配置作为辅助证据。
- 若规则、教程、既有解析和代码存在冲突，优先记录官方规则与当前代码证据，并把冲突列为差异或待确认问题。

## Analysis Structure

目标文档按以下结构组织：

1. 资料与规则依据。
2. 官方比赛规则下的完整比赛流程。
3. 当前代码的整体架构。
4. 当前代码的主运行循环。
5. 完整比赛状态流转过程。
6. 规则状态与代码模块对应表。
7. 各类比赛重启和特殊情况的处理流程。
8. 现有代码解析的核对结果。
9. 已确认实现、间接实现、未找到实现的内容。
10. 存在歧义或需要用户确认的问题。

## Evidence Model

每个关键结论尽量包含：

- 规则来源：官方规则 PDF 中对应章节或可定位描述。
- 代码来源：文件路径、类名、函数名、行为树节点、配置项或关键代码位置。
- 实现分类：明确实现、间接实现、未找到明确实现。
- 状态流转：进入条件、主要处理逻辑、退出或切换条件。

## Compatibility Notes

- 不改变项目结构中的程序代码。
- 若需要新增文档目录，仅新增 Markdown 文档相关路径。
- 不运行自动验证或仿真，最终汇报中明确说明未执行验证。

## Risk Controls

- 将官方教程中的“建议优化”与当前 Demo 的“实际实现”分开记录。
- 对代码未覆盖的规则状态，不以经验补齐；只标记为未找到或待确认。
- 对现有解析文档的判断必须以当前文件证据为准。
