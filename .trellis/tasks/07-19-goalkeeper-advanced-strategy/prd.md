# 守门员动态站位、封堵、出击、解围与回位

## Goal

在不改变现有守门员身份、普通攻防职责和裁判安全分派的前提下，将当前固定站在小禁区中心的 `guard()` 升级为可观察、带迟滞的轻量守门员状态机，使默认或临时守门员能够动态跟球、封堵射门线、安全出击、向前场解围并在动作结束后回位。

## Background

- T01 已建立 STOP/SET 和对方重启安全分派；对方重启使用独立安全走位，不调用普通守门动作。
- T02/T03 已集中选择默认或临时守门员，并在 PLAYING 期间保持临时守门员身份；普通场上职责排除当前守门员。
- T04/T05/T06 已通过统一 NORMAL 入口分派防守、双前场进攻和 `ATTACKING / DEFENDING / CONTESTED` 三态。
- 当前所有普通 live-play 守门入口最终调用 `Player.guard()`；该方法始终前往 `own_goal_area_center()`，并使用 `avoid_ball=True`，可能在门前危险球附近绕球。
- `Context` 已提供 `now`、`dt`、球位置和对手位姿，但 `BallState` 没有速度字段。T06 的 `last_ball_position` 用于丢球恢复，不能直接同时承担可靠逐帧速度采样，因此本任务需要独立的守门员球历史字段。
- 用户允许在 T06 尚未完成手动回归确认的情况下开始 T07；T01-T06 兼容性仍必须由用户最终手动验证。

## Requirements

### R1. 保持守门员身份和裁判安全边界

- 继续复用 T02/T03 选择出的 `current_goalkeeper`，不得重新选择守门员。
- 默认和临时守门员使用相同高级守门逻辑。
- 守门员不得进入 primary attacker、pressure player 或 front partner 候选。
- 对方开球、对方定位球、READY、STOPPED 和无有效裁判数据继续走现有独立安全分支，不进入高级出击或解围逻辑。
- 不可用机器人不得收到守门动作命令。

### R2. 轻量守门员状态机

- 增加 `HOLD / TRACK / BLOCK / CHALLENGE / CLEAR / RETURN` 状态或语义等价的枚举。
- `HOLD`：球未知或远离威胁时守住动态门前目标。
- `TRACK`：球可见但无直接威胁时按球的横向位置小幅跟随。
- `BLOCK`：球快速朝己方球门移动，或球处于直接射门通道且不满足安全出击条件时，优先移动到射门线封堵目标。
- `CHALLENGE`：慢球位于允许出击区域、守门员明显比最近对手先到且场上仍有保护者时主动接近球。
- `CLEAR`：守门员到达门前危险球的解围距离后，向正 X 前场或更空的边路解围。
- `RETURN`：出击条件消失、解围完成或超时后返回动态门前目标。
- 状态切换具有最短保持、不同进入/退出阈值或超时，避免每帧抖动；快速直冲门线的威胁可以立即抢占为 `BLOCK`。

### R3. 动态门前站位

- 基础 X 位于己方门线前方可调安全距离，不再固定使用小禁区几何中心。
- Y 根据球的 Y、跟随增益和球到己方门距离计算，并限制在球门宽度内的安全横向范围。
- 球远时跟随幅度较小；球进入己方后场或接近门前时跟随更积极。
- HOLD/TRACK/RETURN 可避机器人，但 BLOCK/CHALLENGE/CLEAR 不得把危险球作为必须绕开的障碍。

### R4. 独立球速估计

- 在 `store` 中保存守门员策略专用的上一球位置和采样时间，不修改 Framework 协议。
- 仅在球位置和时间有效更新、采样间隔处于合理范围且位移未发生异常跳变时计算简单有限差分速度。
- 对估计速度设置上限；裁判 phase 离开 live play、球丢失或位置跳变时重置历史。
- 不引入卡尔曼滤波或新依赖。

### R5. 射门威胁与封堵

- 速度可靠时，综合球位于己方半场、X 速度朝己方门、投影到门线的 Y 落在球门宽度余量内来判断快速射门威胁。
- 速度不可用时，只能根据球到门距离、中路危险通道和附近对手作保守位置威胁判断，不伪造确定速度结论。
- BLOCK 目标根据球速度射线与门前封堵 X 的交点计算，并限制在守门员安全横向范围。
- 快速来球优先封堵，不为了踢球离开封堵线。

### R6. 安全出击

- 出击至少综合球是否在允许区域、球速是否较慢、守门员到球距离、最近有效对手到球距离、先到优势以及是否存在可用场上保护者。
- 条件至少满足 `goalkeeper_distance + advantage_margin < opponent_distance`。
- 对手位姿全部缺失时不主动扩大出击，只做 TRACK/BLOCK；唯一可用机器人为守门员时同样保守。
- CHALLENGE 超时或条件退出后进入 RETURN，不持续离门追球。

### R7. 安全解围

- 守门员进入可调解围距离后才能 CLEAR。
- 解围方向必须具有正 X 分量，优先前场；根据对手分布选择更空的 `+Y` 或 `-Y` 边路。
- 不向负 X 踢向己方球门，也不选择横穿己方球门中心正前方的目标。
- 解围力度使用集中参数，不散落魔法数字。
- CLEAR 后根据球离开、踢球状态结束或短超时进入 RETURN。

### R8. 可观察性

- action 至少能显示 `goalkeeper:hold / track / block / challenge / clear / return`，并保留默认或临时守门员身份信息。
- DebugDraw 显示动态守门目标；在数据可用时显示射门投影线或解围目标。
- `store` 保存当前守门员模式、进入时间、出击开始时间、威胁原因和最近速度估计，供人工观察。

### R9. 少人和异常降级

- 守门员不可用时继续由 T02 临时守门员机制接管。
- 球未知时回到安全动态门前目标；无法行动或无 pose 时停车。
- 只有守门员一台可用时不主动出击，危险区优先守门和封堵。
- 0 台可用时保持现有直接返回行为。

### R10. 范围与验证约束

- 允许修改 `src/main.py`、`src/player.py`、`src/param.py`。
- 不修改 Framework 数据协议、ROS topic、GameController 解码或 Backend，不引入新依赖。
- 不重构 T04/T05/T06 场上机器人策略，不实现中场固定开球、角球、球门球或界外球。
- 按项目规则不运行 build、test、lint、type-check、format、仿真、开发服务器、watcher 或 IDE 诊断。

## Acceptance Criteria

- [ ] 默认和临时守门员都复用同一 `HOLD / TRACK / BLOCK / CHALLENGE / CLEAR / RETURN` 策略，身份保持逻辑不变。
- [ ] 守门员基础目标位于门线前方，Y 随球横向移动且不越过安全范围；球远和球近使用不同跟随强度。
- [ ] 独立球历史仅在有效采样间隔内产生有限差分速度，异常跳变、球丢失和 phase 变化会重置。
- [ ] 快速朝门且投影进入门框附近的球优先触发 BLOCK，且 BLOCK 不把球作为避障对象。
- [ ] 慢球只有在守门员明显先到、对手位姿有效、出击区域安全且存在场上保护者时才进入 CHALLENGE。
- [ ] 守门员到达危险球后可向正 X 前场或更空边路 CLEAR，不向负 X 或横穿己方门前解围。
- [ ] CHALLENGE/CLEAR 条件结束或超时后进入 RETURN，并回到动态门前目标。
- [ ] 状态具有迟滞、最短保持或超时，不在相邻状态间每帧切换；快速射门威胁可立即进入 BLOCK。
- [ ] action/debug 可观察当前守门员状态、身份、动态目标、威胁或解围目标。
- [ ] T01 对方重启安全、STOP/SET 停车、T04/T05 场上职责、T06 模式切换和临时守门员保持未被改写。
- [ ] 未实现任何固定战术，未运行项目禁止的自动验证命令。

## Out of Scope

- 中场固定开球、角球、球门球和界外球战术。
- 守门员扑倒、跳跃、手部动作或完整物理扑救模型。
- 卡尔曼滤波、轨迹学习、复杂传球规划或完整最优到达时间模型。
- Framework/ROS/GameController 数据协议修改。
- 普通进攻、防守或 T06 模式判断的重构。
