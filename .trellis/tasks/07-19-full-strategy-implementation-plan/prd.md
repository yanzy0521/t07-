# 完整理解策略并制定实现规划

## Goal

完整阅读并解析《3v3 足球机器人比赛整体策略方案-更新版.docx》，结合已完成的比赛规则与当前 Demo 流程分析，在不修改程序代码、不修改项目正式文档、不开始实现的前提下，输出可指导后续逐阶段实现和测试的完整策略实现规划。

## Requirements

- 解析策略文档中的文字和所有战术图片，逐张说明场景、球位置、机器人职责、相对站位、朝向、移动/传球/射门方向、执行顺序、成功处理与失败退出。
- 基于当前项目坐标系，针对我方中场固定开球、我方角球发球、我方球门球发球，给出建议坐标规划、镜像规则、误差范围和待实测标定项。
- 将普通进攻、普通防守、攻防转换、守门员、角色切换、少人/受罚/掉线/摔倒/定位异常、固定战术退出恢复整理为可实现流程。
- 对照当前代码入口、状态机、角色分配、Player 动作和 framework 边界，分析可复用模块、需扩展模块、冲突点和跨帧状态需求。
- 设计总体实现框架，覆盖比赛状态与规则控制、发球权合法行动、在线异常、角色职责、普通攻防、守门员、固定战术、坐标镜像、参数调试等层。
- 分别规划中场开球、角球、球门球状态流程，只设计状态和条件，不写代码。
- 制定支持“小步修改、小步测试”的分阶段优化顺序，每阶段列出目标、文件/模块、逻辑、依赖、风险、测试场景和完成标准。
- 集中列出风险、测试重点和需要用户确认的问题。

## Acceptance Criteria

- [ ] 输出严格包含用户指定的 14 个章节。
- [ ] 所有 DOCX 图片均被逐张查看和解析；若图片无法唯一确定坐标，明确区分图片确定内容、推算内容、实测标定内容和待确认内容。
- [ ] 坐标规划基于当前项目实际坐标系、场地尺寸、球门方向和角度语义。
- [ ] 规划不把普通进攻/防守图片转换成固定绝对站位，而是提出动态计算方式。
- [ ] 对照当前代码说明可复用、需扩展、冲突和跨帧状态需求。
- [ ] 未修改任何程序代码、配置文件、行为树或项目正式文档。
- [ ] 未开始任何策略实现。
- [ ] 未运行 build、test、lint、type-check、format、仿真、部署、开发服务器或其他验证命令。

## Out of Scope

- 编写实现代码。
- 修改 `src/`、配置、行为树或项目正式文档。
- 自动开始第一阶段修改。
- 擅自确定必须实测或用户决策的参数最终值。

## Confirmed Decisions from User

- 中场开球射门队员根据实时位置选择，不固定上下侧。
- 角球接应点和战术站位允许根据对手、球和队友位置动态调整，不完全照抄图片。
- 球门球发球位置不确定具体半场或固定点，可能在罚球区角，应以裁判实际球位和规则安全为准动态计算。
- 中场开球传球基本无误即可射门，整体策略可以更激进。
- 固定战术本地超时初值可采用：开球 4-6 秒，角球 6-10 秒，球门球 5-8 秒。
- 对方重启避让目标距离采用 1.5m。
- 默认守门员优先保留，必要时使用临时守门员。
- 临时守门员可保持到下一次 READY/STOPPED 再交还，避免 PLAYING 中频繁换门将。
- 支援机器人接球后允许直接射门。
- 普通进攻不要求禁止双人追球；允许前场两人参与进攻，避免后场队员完全不参与。
- 允许后续增加固定战术日志字段，例如 tactic_state、locked_roles、exit_reason。
- 参数集中或分散均可，但后续实现应保持可维护和便于调试。
- 重启独占时间和球权开放信任裁判字段，不用本地计时主动开放球权。
- 对方角球防守图作为通用重启防守思路，后续应动态调整，不完全照抄。
- 1-1-1 阵型只适配防守；普通进攻时仍应保证前场 2 人参与。
